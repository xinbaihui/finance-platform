# finance-platform 项目说明

## 1. 产品说明

`finance-platform` 是一个面向个人理财场景的智能产品原型，目标是帮助用户更清晰地管理收入、支出、资产和年度储蓄目标，并通过 AI 对财务数据进行解释与辅助分析。

产品当前由两个核心部分组成：

- `理财副驾驶`：面向普通用户的移动端产品，用于录入个人财务信息、查看年度分析结果，并通过聊天方式获取财务建议。
- `管理后台`：面向运营或演示场景的 Web 管理端，用于查看用户列表、用户财务画像和关键指标，方便进行管理、排查与产品演示。

产品的核心价值主要体现在以下几个方面：

- 把分散的财务数据整合为统一视图，降低用户理解个人收支情况的门槛。
- 将“记录数据”与“解释数据”结合起来，不只是展示数字，还能解释趋势、差距和潜在问题。
- 通过 AI 对年度目标、储蓄达成情况和支出结构进行辅助分析，提升产品的智能化体验。

从用户视角看，这个产品希望回答几个实际问题：

- 我今年挣了多少、花了多少、存下了多少？
- 当前支出结构是否健康，是否存在超支风险？
- 我的年度储蓄目标还差多少，接下来应该优先调整哪些支出？
- 如果收入增加或支出减少，全年结果会发生什么变化？

## 3. 项目技术栈

本项目采用 Monorepo 组织方式，覆盖移动端、Web 管理端、后端服务和数据库建模，整体技术栈如下：

### 3.1 前端技术栈

移动端：

- `React Native`
- `Expo`
- `expo-router`
- `TypeScript`

说明：

- 使用 Expo 进行多端开发与调试，提升开发效率。
- 使用 `expo-router` 管理页面路由，便于组织 Chat、分析、我的等页面结构。
- 使用 `react-native-svg` 绘制图表，用于呈现支出结构等可视化信息。

管理后台：

- `React`
- `Vite`
- `TypeScript`

说明：

- 管理后台以轻量化 Web 应用形式实现，适合快速构建演示型后台系统。
- 使用 React 组件化开发，便于后续扩展用户管理、配置管理和权限管理等后台能力。

### 3.2 后端技术栈

后端服务：

- `Python`
- `Flask`
- `SQLAlchemy`
- `Pydantic / pydantic-settings`

说明：

- 后端负责财务数据聚合、年度分析、上下文构建、聊天接口封装和后台管理接口提供。
- 使用 Pydantic 进行配置管理和数据校验，保证接口输入输出结构清晰可控。

### 3.3 AI 与数据能力

- `OpenAI API`
- `Codex CLI`

说明：

- 聊天能力支持两种 provider：`openai` 和 `codex`。
- 在本地演示模式下，可以通过 `Codex CLI` 直接生成回答；在标准 API 模式下，可以通过 OpenAI 模型完成回复生成。
- 后端会根据用户问题，动态拼装财务上下文，例如财务快照、储蓄目标分析、支出结构分析、全年预测和建议输入，再传递给模型生成结果。

### 3.4 数据库与工程化

- `PostgreSQL`
- `Prisma`
- `pnpm workspace`
- `Nx`

说明：

- Prisma 用于数据库模型定义，当前已建模用户、财务画像、年度计划、交易记录、聊天会话等核心数据结构。
- 使用 `pnpm workspace` 管理多应用依赖。
- 使用 `Nx` 统一组织移动端、后台端和服务端任务，便于在 Monorepo 下进行开发与维护。

## 4. 项目说明

### 4.1 项目定位

`finance-platform` 是一个带有 AI 辅助分析能力的个人理财产品原型项目。它不只是做记账或展示报表，而是尝试把“数据录入”“数据分析”“智能问答”三个动作整合到同一个产品流程中。

项目强调的是产品闭环：

1. 用户先在“我的”页面维护收入、支出、资产和储蓄目标。
2. 系统基于这些结构化数据生成年度财务画像和分析结果。
3. 用户在 Chat 页面用自然语言提问，系统结合财务上下文给出解释、判断和建议。

### 4.2 核心功能

- 用户财务信息维护：支持维护年度收入、年度支出、储蓄目标和资产信息。
- 财务分析看板：支持展示收入、支出、已储蓄金额、目标完成率、支出结构和年度预测。
- AI 理财问答：支持围绕“本月分析”“全年预测”“目标差距”等主题进行提问。
- What-if 场景分析：支持对“如果收入增加”“如果支出减少”等自然语言问题进行简单模拟。
- 后台管理能力：支持查看用户列表、用户详情及关键财务指标，便于演示和管理。

### 4.3 设计与实现思路

这个项目的实现重点不只是页面开发，而是“业务建模 + AI 上下文组织”的结合。

在业务侧，项目将用户财务数据拆分为几个更容易分析的维度：

- 收入
- 支出
- 资产
- 年度目标
- 聊天上下文

在 AI 侧，系统不会把所有数据直接原样丢给模型，而是先基于用户问题选择相关分析模块，再构建结构化上下文。例如：

- 当用户问“目标差距”时，优先注入储蓄目标和全年预测相关数据。
- 当用户问“本月支出”时，优先注入支出结构和财务快照。
- 当用户问“如果我少花 2000 会怎样”时，会先解析自然语言中的变量，再进行模拟分析。

这种方式的好处是：

- 能减少无关上下文，提高回答的针对性。
- 能让回答更贴近真实业务逻辑，而不是泛泛而谈。
- 为后续进一步做规则引擎、推荐策略或更细粒度的 AI Agent 能力预留空间。

### 4.4 项目亮点

- 以真实业务场景为核心，而不是单纯做一个聊天 Demo。
- 同时覆盖移动端、后台端、后端服务和数据库建模，体现了完整产品链路能力。
- 将结构化财务分析与自然语言交互结合，体现了 AI 在垂直业务中的落地思路。
- 使用 Monorepo 统一管理多个应用，体现了工程组织能力和项目扩展性。

- 后端如何为 AI 问答构造财务上下文，提高回答质量。

- 在 Monorepo 架构下，如何组织前端、后端和数据库模块，保证开发效率。

`finance-platform` 是一个以个人理财为主题、以 AI 辅助分析为亮点的全栈项目。它既展示了产品设计思路，也体现了前端开发、后端接口设计、数据库建模和 AI 集成能力。对于面试场景而言，这个项目不仅能展示技术实现，还能体现对业务理解、产品意识和系统化设计能力。
